# 需求设计文档 — IP 异常交易管控规则体系

## 原始需求描述
每天都有多个内部风控交易员监控盘面，目前极度依赖延时的外包数据库查询来抓取：（1）同IP跨账号对冲、（2）黑名单客户试探、（3）云主机/IDC 伪造属地 IP 的违规薅羊毛行为。由于存在时差和非实时性，导致公司持续性“失血”亏损。
目标是利用 Manager API 的实时性，在这套即将商业化 SaaS 出售的风控后台上，增添这三种自动报警和拦截闭环机制。并且首期交付的切入点定为最基础的 Black List（黑名单）。

## 问题重定框
你以为你需要的是一个能记录比对 IP 黑名单、多账号对冲和虚假数据中心IP的自动化过滤列表；但实际上，你要解决的是**如何把人工被动查单变为“自动化交易阻拦网”**，并且这层网络必须具备极强的**“SaaS 多租户界面化配置能力”**，让任意一家新客户接入后，他们的风控员都能在 1 分钟内配好本公司的阻断策略并立刻生效。

## 关键假设与挑战
- **[假设 1]**：你假设交易员仅仅是在 Microsoft Teams 里收到一个通知就完事了。
- **[挑战]**：如果目标是商业化的 SaaS，除了发发通知，必须在系统的 Alert 面板内给交易员配备“一键核查确诊 / 直接拉入黑名单库”的界面闭环。
- **[假设 2]**：你假设依赖“黑名单匹配”能一劳永逸。
- **[挑战]**：黑产羊毛党如今高频使用动态代理 IP。导致静态的 Black list 防御极易被绕过；所以“Fake IP”（基于属地环境行为）和“Hedge”（跨账号异常交易关联）这两种**行为模式判别**在未来攻防中反而能起到决定性作用。

## 三种实现路径

| 路径 | 核心思路 | 时间估算 | 主要风险 |
|------|---------|---------|---------|
| **最小路径（验证假设）** | 依托现有原型，仅打通 **黑名单 (Black List)** 的前端配置界面（录入 CID / IP / 拦截原因），并接入模拟警报列表。 | 1 天 | 防御面相对窄，仅能拦截已知黑产；但对于跑通“配置->报警->拦截”的主干线是最佳选择。 |
| **标准路径（功能完整）** | 在黑名单基础上，增设 **Fake IP**（基于地区与 IDC/代理检测）与 **Position Hedge（对冲）**（相同 IP 下的反向订单监控）规则看板与阈值表单。 | 2-3 天 | Fake IP 判定未来需购买/对接靠谱的高端外部 IP 指纹库接口；对冲算法需处理较大数据量并发。 |
| **完整路径（长期愿景）** | 在 SaaS 管理端增加“IP 串联关联图谱”，无需人工分析，一旦某一节点疑似刷单，系统直接冻结该团伙整个交易链路。 | 1-2 周 | 复杂度飙升，存在误杀大客户的可能，初期难以实施。 |

## 推荐方案
**极力推荐：最小路径（先跑通黑名单闭环） 叠加 标准路径的原型界面展示。**
理由：遵照你的思路，从 Black List 切入能最快让业务方看到“疗效”。我们将基于之前已开发的 `trading-risk-admin/index.html` 以及 `router.js` 原型，向侧边栏新增这三个 IP 规则模块的设置面板。
