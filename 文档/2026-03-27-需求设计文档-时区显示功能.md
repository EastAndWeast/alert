# 需求设计文档 — 告警时区显示功能

## 原始需求描述

客户反馈告警记录里的触发时间均显示为 UTC+0，无法与 MT5/MT4 服务器的本地业务时间对应。  
内部已讨论两种方案：方案一为显示 UTC+0 与服务器本地时间两个字段；方案二由 C++ 端直接传服务器时间（已否决，灵活性差）。

## 问题重定框

> 你说想解决"时间显示是 UTC+0"的问题。
> 但你实际上在解决的问题是：**一家公司的多台不同时区服务器的告警，混在同一个列表里时，风控人员无法快速判断某条告警发生在该服务器的"什么业务时间"，导致事件响应变慢且混乱。**

## 关键假设与挑战

**[假设 1]**：用户看到 UTC+0 时间后会自己换算服务器本地时间。  
**[挑战]**：用户不知道每台服务器是哪个时区，根本无从换算，更遑论两台时差 16 小时的服务器混排显示。

**[假设 2]**：只要显示了服务器时间，用户的问题就解决了。  
**[挑战]**：时间筛选框如果仍按 UTC 筛选，而用户用服务器时间来思考，会造成筛选结果与预期不符的新困惑——需要在 UI 上明确标注筛选基准。

**[假设 3]**：C++ 直接传服务器本地时间是最省事的方案。  
**[挑战]**：不同客户的服务器时区不同，定死在 C++ 层会导致数据失去统一基准，前端无法灵活按 UTC 排序，后续报表、日志也会混乱。已否决，正确方向是传时区差值。

## 三种实现路径

| 路径 | 核心思路 | 时间估算 | 主要风险 |
|------|---------|---------|---------|
| **最小路径** | 在告警时间列加一行"服务器时间"的子文本，纯前端换算 | 1天 | 列宽变窄，两行不利于筛选 |
| **标准路径（推荐）** | 新增"服务器时间"独立列，可在列设置中开关；筛选仍基于 UTC；Java 返回时区差值字段 | 3-5天 | Java 需小幅改造 |
| **完整路径** | 标准路径 + 全局时区偏好设置；推送通知（Telegram/Lark）同步加服务器时间；告警时间筛选支持服务器时间模式 | 2-3周 | 多端联动改造，测试面广 |

## 推荐方案：标准路径

**推荐理由**：
- 数据库已有服务器时区字段，Java 改造量小（仅需在告警 API 返回中加入 `timezone_offset` 字段）。  
- 告警列表已有列显示/隐藏功能，新增一列复用现有机制，改动极小。  
- 不影响 UTC 排序和筛选逻辑的准确性，技术风险低。

**具体实现说明**：

### 数据层（Java）
在告警列表 API 的每条记录中新增两个字段：
```json
{
  "triggered_at": "2024-03-27T14:32:05Z",      // 已有，UTC 时间戳
  "server_timezone_offset": -300,               // 新增，分钟数（UTC-5 = -300）
  "server_name": "Alpha-US"                     // 已有，数据源名称
}
```

### 前端展示（Web 告警列表）
新增「服务器时间」列，计算逻辑：
```
服务器时间 = UTC时间 + timezone_offset（分钟）
显示格式   = "09:32:05 (UTC-5)"
```

列设置控制（可选显示）：
```
[✓] 告警时间 (UTC)        ← 默认显示，用于排序/筛选
[✓] 服务器时间            ← 默认显示，可关闭
```

**筛选说明（需在 UI 上标注）**：
> ℹ️ 时间筛选基于 UTC+0，与告警时间列一致

### 推送通知（可选，第二期）
在 Telegram / Lark / 邮件模板中补充服务器时间字段：
```
⚠️ 大额交易告警
时间: 14:32 UTC+0  |  09:32 (Alpha-US / UTC-5)
```
