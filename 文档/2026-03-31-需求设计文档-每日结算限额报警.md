# 需求设计文档 — DSL Limit (Daily Settlement Limit) 日限额报警

## 原始需求描述
客户希望上线 **DSL Limit (Daily Settlement Limit Alerts)** 功能。
期望效果：当某个产品的 Day PnL（单日盈亏）达到设定的阈值参数（如盈利达到 +5,000 或 亏损达到 -10,000 ），系统自动发出警报。
算法公式：
- `Day Pnl = -1 * Closed Pnl + Change In Floating Pnl`（当天的已平仓的盈亏 + 未平仓的浮动盈亏变化）
- `Change in Floating Pnl = 前一天EOD时候的浮动盈亏 - 当前的浮动盈亏`

## 问题重定框
你说你想做 **DSL Limit（日常超限报警器功能）**，但你实际上在解决的问题是：**将后台 3-5 人团队从每天导出数据、跨服务器汇总计算盈亏这种存在致命“延迟风险”的手工体力活中解放出来，把事后被动结账转变为“事前/事中实时程序穿透并亮灯”的防漏闭环。**

## 关键假设与挑战
- **[假设 1]**：你假设目前的计算公式只要放到警报器里就万事大吉了。
- **[挑战]**：跨不同服务器（Server A/B...）实时计算前一天 EOD 的相对变化值并加总，这不是轻量级的判断。警报系统在前端只能提供参数控制配置，其背后的风控引擎必须具备全时全盘的聚合算力。
- **[假设 2]**：你假设 +5000 / -10000 这样的参数只要输进去就行。
- **[挑战]**：交易产品的资金敞口属性不同，像黄金(XAUUSD) 亏 10000 刀和一些小市值的流动性差的加密货币亏 10000 刀的概念完全不同。系统必须支持在创建规则时绑定具体的**产品过滤器 (Symbol Filters)**。

## 三种实现路径

| 路径 | 核心思路 | 时间估算 | 主要风险 |
|------|---------|---------|---------|
| **最小路径（验证假设）** | **(推荐)** 优先在此风控管理后台（`trading-risk-admin`）提供完整的可读界面。风控人员能像配置其它大单监控一样，在 UI 上选产品、定盈亏红线，并实时从底部的历史列表中直观捕捉异常 PnL 数据。 | 0.5 天 | 需验证后端真正下推的计算速度是否能做到低延迟。 |
| **标准路径（功能完整）** | 在提供界面参数的基础上，增加服务器级、组级别的独立规则配置。后台自动向第三方通讯工具（如 Telegram/Teams）发送汇总排行榜报警。 | 2-3 天 | 后端架构的数据链路集成压力变大，报警噪声如未归并会造成骚扰。 |
| **完整路径（长期愿景）** | 建设实时的『全盘资产看板』(Dashboard View)。除了红线警报，提供一张可视化的大盘数据表，每个产品的实时 Day PnL 柱状起伏一目了然。 | 一周以上 | 性能挑战极高，重度依赖 Manager API 发射频率。 |

## 推荐方案
**推荐：最小路径（优先配置层打通）**。
- 理由：由于客户“已通过 Manager API 抓到了数据、且无技术限制”，因此最紧迫的是快速在 SaaS 软件中为这一核心风控逻辑搭建一个**高颜值的规则表单与实时捕获记录表**，供风控直接上手替换掉之前的 Excel。
