## 一、数据流图

```
【Global 管理员】
    │
    ├─ 配置发件服务（SMTP/API 凭证）
    │       ↓
    │  [email_services 表]
    │       ↓
    ├─ 分配给子公司（company.email_service_id）
    │
【子公司管理员】
    │
    ├─ 开启/关闭公司邮件功能（company.email_enabled）
    ├─ (业务端默认从全局邮件通道发出，无需修改各规则参数)
    │
【子公司用户/账号】
    │
    ├─ 设置通知邮箱（user.notification_email，默认=login email）
    └─ 开启/关闭自己的邮件通知（user.email_notify_enabled）

【告警触发时】
规则引擎检测到告警
    ↓
查询该公司 email_enabled == true 且绑定了 email_service
    ↓
查询该公司下所有 email_notify_enabled == true 的用户
    ↓
逐一通过绑定的发件服务 API/SMTP 发送邮件
    ↓
记录发送状态（成功/失败）
```

## 三、状态机图

### 3.1 发件服务生命周期

```
[新建] 
  → (配置填写完成 + 测试发送成功) → [active/可用]
  → (测试发送失败) → [error/配置错误]
  → (手动停用) → [inactive/停用]

[active/可用]
  → (编辑后测试失败) → [error/配置错误]
  → (手动停用) → [inactive/停用]

[error/配置错误]
  → (修复配置并重新测试成功) → [active/可用]

[inactive/停用]
  → (手动启用) → [active/可用]
  → (删除) → [已删除]
```

### 3.2 用户邮件通知状态

```
[账号创建]
  → notification_email = 登录邮箱 (默认填入)
  → email_notify_enabled = false (默认关闭)

[用户开启邮件]
  → email_notify_enabled = true
  → 所有 Email 推送的规则触发时，该用户将收到邮件

[用户修改通知邮箱]
  → notification_email = 新邮箱
  → 下一次告警将推送至新邮箱

[公司管理员关闭公司邮件]
  → company.email_enabled = false
  → 即使用户 email_notify_enabled = true，也不会发送（公司级开关优先）
```

### 5.1 数据存储方案

**结论：复用 MockData 内存对象，新增 `email_services` 数组，扩展 `companies` 和 `users` 对象字段。**

新增 Mock 数据结构：

```javascript
// MockData 新增字段
email_services: [
  {
    service_id: 'ES001',
    name: 'SendGrid Production',
    provider: 'sendgrid',      // sendgrid | mailgun | ses | postmark | smtp
    status: 'active',          // active | inactive | error
    api_key: 'SG.xxx...',      // provider=api类型时
    smtp_host: '',             // provider=smtp时
    smtp_port: 587,
    smtp_user: '',
    smtp_pass: '',
    smtp_tls: true,
    created_at: '2026-03-27'
  }
]

// companies[] 扩展
{
  ...existing_fields,
  email_enabled: true,
  email_service_id: 'ES001',
  from_name: 'Alpha Broker Alerts',
  from_email: 'noreply@alpha.com'
}

// users[] 扩展
{
  ...existing_fields,
  notification_email: 'admin@alpha.com',  // 默认 = email字段
  email_notify_enabled: false
```

### 5.2 API 设计（前端模块接口）

**结论：纯前端 Mock 实现，遵循现有 CRUD 模式，无真实网络请求。**

| 功能 | MockData 操作 | 对应前端入口 |
|------|-------------|------------|
| 列出发件服务 | `MockData.email_services` | Global 设置 → 发件服务列表 |
| 新增发件服务 | `addEmailService(data)` | 点击"新增"弹窗 |
| 更新发件服务 | `updateEmailService(id, data)` | 点击"编辑"弹窗 |
| 删除发件服务 | `deleteEmailService(id)` | 删除前检查是否被公司引用 |
| 公司分配服务 | `companies[i].email_service_id = id` | 公司管理 → 编辑 |
| 用户设通知邮箱 | `users[i].notification_email = email` | 用户管理 → 编辑 / 个人设置 |
| 用户开关邮件 | `users[i].email_notify_enabled = bool` | 用户管理 → 编辑 / 个人设置 |
| 测试发送 | 前端模拟：校验字段非空后 showToast | 各配置页的"测试"按钮 |

---

### 5.3 状态管理

**结论：全部状态驻留在 MockData 内存中，与现有其他模块一致。无需引入新的状态同步机制。**

- 客户端状态：表单临时状态（用户填写过程中未保存）
- 服务端状态（MockData）：`email_services[]`、`companies[i].email_*`、`users[i].email_*`
- 无需额外同步策略，刷新即 reset（Mock 环境特性）

---

### 5.4 失败模式

| 失败类型 | 处理策略 |
|---------|---------|
| **发件服务填写有误** | 点击测试时前端校验 + 模拟失败 toast，不允许保存 status=error 的服务 |
| **删除了仍在使用的发件服务** | 删除前检查 `companies[].email_service_id`，若有引用则提示并阻止删除 |
| **公司未分配发件服务就开启邮件** | 保存时校验，提示"请先配置发件服务再开启" |
| **通知邮箱格式非法** | 前端 regex 校验，不通过禁止保存 |
| **规则推送时发件服务不可用** | 模拟：记录发件失败日志，告警记录本身仍正常保存和显示 |

---

### 5.5 信任边界

| 数据 | 可信程度 | 验证方式 |
|------|---------|---------|
| Global 管理员填写的 SMTP/API 配置 | 中（内部人员但可能误填）| 通过测试发送验证 |
| 子公司管理员填写的通知邮箱 | 低（可能填错别人邮箱）| 格式校验 + 依赖用户自测 |
| 普通账号填写的通知邮箱 | 低 | 同上 |
| 发件服务 API Key | 高敏感度 | 前端存储为 password 类型 input，后端加密存储（真实环境必须）|

**权限边界（角色控制）：**
- `manage_settings` 权限：才能访问 Global 发件服务管理
- `manage_companies` 权限：才能编辑公司的邮件服务分配
- `manage_users` 权限：才能编辑其他人的通知邮箱
- 普通用户：只能修改自己的通知邮箱和开关

---

### 5.6 性能与扩展

**结论：当前 Mock 阶段无性能问题。真实环境注意以下扩展点。**

| 场景 | 规模估算 | 扩展建议 |
|------|---------|---------|
| 单次告警触发邮件数 | 每公司约 5-20 人 | 小批量，同步发送即可 |
| 发件服务数量 | Global 维护约 5-20 个 | 无需分页，列表展示 |
| 告警高峰并发 | 极端情况多规则同时触发 | 真实环境需引入发送队列（Redis/MQ）|
| 多服务商切换 | 工厂模式已支持 | 新增服务商只需新增 Adapter |

---

### 5.7 安全考虑

| 风险 | 措施 |
|------|------|
| API Key 明文存储 | Mock 阶段暂以 password input 隐藏；真实环境必须后端加密存储 (AES/Vault) |
| 跨公司数据隔离 | Global 管理员才能看到所有发件服务；子公司只能看到被分配的服务名称，看不到凭证 |
| 垃圾邮件风险 | 建议真实环境配置 SPF/DKIM/DMARC 记录；告警限流防刷 |
| 任意邮箱注入 | 通知邮箱必须前端格式校验（regex）+ 后端二次校验 |

## 六、技术决策清单

**[决策 1]：邮件配置层级 — 公司级还是全局级？**
> **推荐：Global 统一管理发件服务，公司级分配使用。**
> 理由：凭证集中管理安全性更高；子公司无需接触 API Key；未来计费按公司维度统计邮件量更方便。

**[决策 2]：收件人模型 — 账号级还是规则级？**
> **推荐：账号级（每人有自己的通知邮箱和开关）。**
> 理由：与 Web 渠道对齐；避免管理员需要为每条规则维护收件人列表；账号自己决定是否接收，更灵活。

**[决策 3]：发件服务集成方式 — 统一 SMTP 还是分别对接各家 API？**
> **推荐：工厂模式，同时支持 SMTP 和各家 HTTP API。**
> 理由：SMTP 是万能兼容方案（所有服务商都支持）；API 方式发送成功率更高且支持发送日志追踪。初期实现 SMTP + SendGrid 即可覆盖大多数需求。

**[决策 4]：测试发送 — Mock 还是真实发送？**
> **推荐：前端 Mock 环境模拟测试（校验字段格式后 toast 提示）；后端真实实现时接真实 API。**
> 理由：当前为纯前端 Demo，保持一致性。

**[决策 5]：前端入口 — 新增独立菜单还是整合进现有菜单？**
> **推荐：不新增菜单，整合进现有模块。**
> - Global 发件服务 → 全局配置（Settings）页中新增一个 section
> - 公司邮件分配 → 公司管理编辑弹窗中新增字段
> - 用户通知邮箱 → 用户管理编辑弹窗 + 个人设置页（Profile）
