## 2.1 数据流图

```
单条克隆流程：
用户点击规则卡片"克隆"按钮
  → 读取当前规则数据（rule object）
  → 读取可用服务器/数据源列表（datasources）
  → 弹出克隆预览弹窗（左：原始只读 | 右：可编辑克隆数据）
  → 自动生成新规则名称 "[原名] - [目标服务器名]"
  → 用户在右侧确认/修改字段（重点：symbol_filter / group_filter 置空）
  → 用户点击"确认克隆"
  → 执行深拷贝：新建 rule 对象（新 rule_id，更新 source_id, custom_name）
  → 写入 MockData.rules（生产环境对应 POST /api/rules）
  → 弹窗关闭，刷新规则列表，Toast 提示"克隆成功"

批量（一键）克隆流程：
用户点击"📋 克隆到服务器"按钮
  → 读取当前数据源下全部规则（N条）
  → 弹出目标服务器选择器
  → 用户选择目标服务器后，进入方案A批量预览弹窗：
    - 顶部：共 N 条规则将克隆至 [目标服务器]
    - 列表：每条规则折叠卡片，点击可展开左右对比
  → 批量生成新规则名称（逐条应用命名算法）
  → 用户点击"全部确认克隆"
  → 逐条执行深拷贝，写入 source_id = 目标服务器ID
  → 弹窗关闭，Toast 提示"成功克隆 N 条规则"
```

## 2.4 测试矩阵

| 测试场景 | 正常路径 | 边界条件 | 异常路径 |
|---------|---------|---------|---------|
| **单条克隆** | 选目标服务器 → 预览 → 确认 → 新规则出现在目标服务器列表 | 目标服务器 = 来源服务器（克隆到自身） | 来源服务器已被删除（规则为孤儿规则） |
| **批量克隆** | N 条规则全部克隆成功，Toast 显示"成功 N 条" | N = 0（当前服务器无规则）→ 按钮应置灰 | 目标服务器列表为空 |
| **命名算法** | "大单监控" + "MT5Demo" → "大单监控 - MT5Demo" | 无自定义名称时使用规则类型 | 同名规则重复克隆 → 追加 "(2)" |
| **Group/Symbol 为空** | 有 symbol_filter 的规则，目标服务器不存在该 Symbol → 置空 | symbol_filter 本来就是空 → 克隆后保持空 | symbol_filter 为非法值 |
| **权限验证** | 有 manage_rules 权限的用户可操作 | 只读用户不显示克隆按钮 | 低权限用户通过接口直接调用 → 返回 403 |
| **名称重复序号** | "大单监控 - MT5Demo" 已存在 → 生成 "(2)" | 已存在 "(2)" → 生成 "(3)" | 连续克隆 100 次序号累计 |

## 技术维度审查

### 1. 数据存储方案

**结论（明确）**：当前系统为 SPA + MockData，无持久化数据库，规则存储在内存数组 `MockData.rules[]` 中。

克隆操作的数据模型：
```javascript
// 原规则对象结构（从 rules.js saveRule 逻辑推断）
{
  id: 'R_001',             // 系统唯一 ID（克隆时生成新 ID）
  rule_type: 'scalping',   // 规则类型（克隆时保持不变）
  source_id: 'DS_001',     // 来源数据源/服务器（克隆时改为目标）
  custom_name: '大单监控',  // 自定义名称（克隆时自动生成新名）
  enabled: false,          // 克隆后默认禁用（避免意外触发）
  params: { ... }          // 各规则的业务参数（除 symbol_filter/group_filter 外全量复制）
}
```

> **关键决策**：克隆后的规则 `enabled` 默认设为 **false（禁用）**，需用户手动确认开启，防止规则立即在未审查的情况下触发告警。

### 2. API 设计（面向未来后端）

**结论（明确）**：当前前端，未来对接后端时建议以下 API 设计：

```
POST /api/rules/clone
Body: {
  source_rule_id: "R_001",
  target_source_id: "DS_002",
  overrides: {              // 用户在右侧编辑的覆盖值
    custom_name: "大单监控 - MT5Demo",
    params: { symbol_filter: [] }
  }
}
Response: {
  rule_id: "R_099",         // 新规则 ID
  created_at: "2026-03-27T..."
}

POST /api/rules/batch-clone
Body: {
  source_source_id: "DS_001",  // 来源数据源（全量克隆该源下全部规则）
  target_source_id: "DS_002",  // 目标数据源
  rule_overrides: [{ ... }]    // 可选：对具体某条规则的字段覆盖
}
Response: {
  cloned: 8,
  failed: 0,
  rule_ids: [...]
}
```

### 3. 状态管理

**结论（明确）**：

| 状态 | 存储位置 | 说明 |
|------|---------|------|
| 克隆弹窗开关状态 | 前端 DOM（Modal 显隐） | 无需持久化 |
| 用户在右侧的编辑值 | 弹窗内 JS 对象（临时） | 取消后丢弃，确认后写入规则 |
| 目标服务器选择 | 弹窗内 JS 变量 | 临时 |
| 规则列表数据 | `MockData.rules[]`（内存） | 克隆成功后追加新规则，触发页面刷新 |

### 4. 失败模式

| 失败模式 | 处理策略 |
|---------|---------|
| **目标服务器列表加载失败** | 下拉框显示"暂无可用服务器"，按钮置灰，不弹窗 |
| **命名冲突（连续克隆）** | 客户端命名算法自动追加序号 `(N)`，不阻断流程 |
| **克隆到自身（source = target）** | 目标服务器下拉框中禁用当前服务器选项，从根源避免 |
| **批量克隆中途失败（未来后端）** | 记录成功/失败条目，弹窗展示汇总，已成功的不回滚（幂等设计） |
| **规则 params 中含未知字段** | 深拷贝不做字段过滤，原样复制，保持前向兼容 |

### 5. 信任边界

**结论（明确）**：

- **可信输入**：当前 SPA 环境下，MockData 为受信任的数据源，无需额外验证。
- **权限边界**：
  - 前端隐藏克隆按钮（`isReadOnly` 检查）
  - 生产环境后端侧验证 `manage_rules` 权限，防止越权调用
  - 用户只能克隆自己有权限访问的数据源下的规则（`getUserSourceIds` 过滤）
- **跨租户隔离**：克隆操作只能在当前用户权限范围内的 `source_id` 之间进行，不能克隆跨公司（`company_id`）的规则。

### 6. 性能与扩展

**结论（明确）**：

- **当前规模**：每个服务器规则数量预计 10-50 条，批量克隆单次最大 50 条，无性能问题。
- **潜在瓶颈**：若未来规则数量超过 500 条，批量预览弹窗一次性渲染所有卡片会有卡顿，届时需引入虚拟列表（Virtual Scroll）。
- **扩展方案**：API 层面 batch-clone 支持 `dry_run` 参数，先预览影响范围，再实际执行。

### 7. 安全考虑

- **当前（Demo 阶段）**：纯前端，无安全风险。
- **生产环境**：
  - 克隆接口需 JWT 认证 + `manage_rules` 权限校验
  - `source_rule_id` 必须属于当前用户有权访问的数据源（后端二次验证）
  - 日志记录：克隆操作写入操作审计日志（operator, action, timestamp, source_id, target_id）

## 技术决策清单（已全部确认）

**[决策 1]**：克隆后启/禁状态控制方式？
**已定：弹窗内提供「全选启用 / 全选禁用」开关 + 每条规则单独勾选**
理由：给用户充分控制权，不强制任何默认行为，用户在预览弹窗里亲自决定每条规则的初始状态。

**[决策 2]**：超出长度的规则名如何处理？
**已定：截断至 50 字符，末尾显示 "..."**

**[决策 3]**：是否支持同时克隆到多个服务器？
**已定：不支持，每次只能选一个目标服务器**
操作多服务器时，用户重复执行即可。

**[决策 4]**：批量克隆时各条规则命名是否独立？
**已定：完全独立，各条规则按"原规则名 + 目标服务器名"生成，互不影响**

**[决策 5]**：批量克隆高危操作的确认机制？
**已定：三步流程 + 输入目标服务器名作为最终确认（不要求密码）**

```
批量克隆三步确认流程：

步骤 1 → 选择目标服务器 + 设置全局启用/禁用开关
步骤 2 → 方案A逐条预览弹窗（展开可单条调整右侧字段）
步骤 3 → 二重确认弹窗（防误操作）：
  ┌──────────────────────────────────────────────┐
  │ ⚠️ 请确认克隆操作                               │
  │                                              │
  │ 即将克隆 N 条规则到 "MT5Demo"                  │
  │ 请在下方输入目标服务器名称以确认：               │
  │                                              │
  │ [ 输入框：MT5Demo                    ]        │
  │                                              │
  │              [取消]  [我确认，开始克隆]          │
  └──────────────────────────────────────────────┘
```

> 不要求密码的原因：此操作每周发生，密码输入摩擦极大。输入目标服务器名既防误触，又让用户明确知道自己在做什么。
>
> 单条克隆无需三步确认——已有「预览 → 确认」两步足够。
>
> **后续规划建议**：规则列表支持多选 + 批量删除，从根本上解决"万一克隆多了不好清理"的问题，比加重克隆门槛更治本。
